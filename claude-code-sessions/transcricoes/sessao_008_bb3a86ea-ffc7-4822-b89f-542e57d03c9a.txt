**Teste Principal - Cilindro vs AABB (Linhas 2003-2017):**
```cpp
bool CylinderIntersectsAABB(glm::vec2 pos_xz, float raio, float y_min, float y_max, const CollisionAABB&amp; box)
{
    // 1. Overlap em Y
    if (y_max &lt; box.min.y || y_min &gt; box.max.y) return false;
    
    // 2. Distância do círculo ao retângulo (projeção XZ)
    float closest_x = glm::clamp(pos_xz.x, box.min.x, box.max.x);
    float closest_z = glm::clamp(pos_xz.y, box.min.z, box.max.z);
    float dx = pos_xz.x - closest_x;
    float dz = pos_xz.y - closest_z;
    
    return (dx*dx + dz*dz) &lt; (raio * raio);
}
```

**Teste com Paredes Diagonais (Linhas 2019-2047):**
```cpp
bool CylinderIntersectsLine(glm::vec2 pos_xz, float raio, float y_min, float y_max, const CollisionLine&amp; line)
{
    // Produto vetorial: projeta cilindro sobre a parede diagonal
    glm::vec2 ab = line.p2 - line.p1;
    glm::vec2 ap = pos_xz - line.p1;
    float t = glm::clamp(glm::dot(ap, ab) / glm::dot(ab, ab), 0.0f, 1.0f);
    glm::vec2 closest = line.p1 + t * ab;
    
    return glm::dot(pos_xz - closest, pos_xz - closest) &lt; (raio * raio);
}
```

```cpp
{

    // Eixo X
    glm::vec3 candidate_x = Pos_Player + (dx, 0, 0);
    if (!PlayerCollidesAt(candidate_x)) {
        Pos_Player.x = candidate_x.x;
    } else if (IsPlayerOnGround()) {
        // Tenta subir um degrau
        glm::vec3 step = Pos_Player + (dx, MAX_STEP_HEIGHT, 0);

    // Análogo para Eixo Z (deslizamento em paredes)
}
```

Ótimas perguntas. Deixa eu explicar a diferença das duas técnicas de forma concreta, e depois respondo sobre a migração do snap.

## A pergunta-chave das duas técnicas
Em ambas, a ideia é a mesma: criar uma **câmera virtual** posicionada "atrás" do portal de destino (a posição da sua câmera real, transformada pela matriz portal→portal) e renderizar a cena por ela. A diferença é **onde** essa segunda renderização vai parar e **como** ela aparece dentro do buraco do portal.

### Opção A — Stencil buffer
O framebuffer ganha um canal inteiro extra por pixel (o "stencil"), como uma máscara de recorte.
1. Desenha a cena normal na tela.
2. Desenha o **quad do portal só no stencil** (sem cor), marcando "estes pixels são a janela do portal".
3. Re-renderiza a cena inteira pela câmera virtual, mas o OpenGL **só deixa escrever onde o stencil está marcado**. Resultado: o outro lado aparece recortado exatamente no formato do portal, na mesma imagem.
- Para não desenhar o que está *entre* a câmera virtual e o portal de destino (que tamparia a vista), inclino o plano *near* da projeção para coincidir com o plano do portal — isso é feito **editando a matriz de projeção** (a mesma da aula), sem tocar no shader.

**Prós:** pixel-perfect, mesma resolução da tela automaticamente, **zero gestão de texturas ou resize**, recursão é natural (basta incrementar o valor do stencil), é literalmente a técnica do Portal original e usa exatamente as matrizes `view`/`projection` da aula.
**Contras:** as operações de stencil são um pouco mais sutis de entender; a janela precisa ter stencil bits (o GLFW já dá 8 por padrão — confirmo isso no setup).

### Opção B — FBO (render-to-texture)
Crio um framebuffer fora-da-tela (FBO) com uma textura própria.
1. Renderizo a cena pela câmera virtual **dentro dessa textura** (fora da tela).
2. Renderizo a cena real na tela e, ao desenhar o quad do portal, **amostro essa textura** usando a posição de tela do fragmento (como uma "TV mostrando a outra sala").

**Prós:** fácil de depurar (dá pra mostrar a textura na tela e ver o que foi renderizado); mentalmente simples ("é uma TV").
**Contras:** preciso criar e gerir FBOs, **recriá-los a cada resize da janela**, perda de qualidade se a resolução da textura ≠ tela, e um ramo de `sampler` no fragment shader; recursão precisa de "ping-pong" entre 2+ FBOs (mais chato que o stencil).

**Minha recomendação: Stencil.** Para 2 portais com recursão de 1 nível, é mais limpo, sem dor de cabeça de resize, pixel-perfect e casa com o material da aula.

## Sobre migrar SNAP → clone/straddle no futuro
Dificuldade **média, e majoritariamente aditiva** — não é um rewrite. O motivo: tanto o snap quanto o clone precisam da **mesma matriz de transformação portal→portal**. # Plano: Implementação de Portais (Portal-3)

## Context

O projeto `Portal-3` é o trabalho final de CG da UFRGS (dupla Gabriel Knopp + Tobias),
uma reimplementação inspirada em *Portal* em C++/OpenGL. A `SPEC.md` já prevê portais como
peça central ("Projeções para a renderização dos portais — exemplo da aula de projeção" e
"ao criar um novo portal numa nova localização, a instância antiga é removida"). Hoje o jogo
tem salas, jogador FPS, caixa, rádio, botão/porta e câmeras — **mas nenhum portal**.

Precisamos implementar portais com três features: (1) **see-through** (olhar por um portal
mostra a vista do outro), (2) **transformação linear** dos vetores físicos na travessia
(continuidade de movimento, mesmo com portais **não paralelos às paredes**), e (3) uma
**API de instanciação intuitiva** para o desenvolvedor. Resultado esperado: atravessar um
portal teleporta jogador/entidades para o portal-par com continuidade visual e física.

### Decisões já travadas com o usuário
- **See-through:** **Stencil buffer** (recorta a vista da câmera virtual no buraco do portal,
  no mesmo framebuffer; pixel-perfect; sem gestão de FBO/resize; usa as matrizes da aula).
- **Recursão:** **1 nível** (olhar por um portal mostra a outra sala; portais lá aparecem com fundo).
- **Travessia:** **SNAP** quando o centro da entidade cruza o plano do portal. O design deve
  **deixar aberto** o caminho para "clone/straddle" no futuro → a matriz de transformação
  portal→portal será pública e a detecção de travessia ficará isolada.

## Restrições técnicas
- Build com **`-std=c++11`** (`Makefile.macOS`) — sem recursos de C++14/17.
- Um único shader program (`g_GpuProgramID`); objetos discriminados por `object_id` no
  fragment shader.
- Convenção de câmera UFRGS (olha em −z; near/far negativos). `Matrix_Perspective` retorna `-M*P`.

## Arquitetura — novo módulo desacoplado

Criar **`include/portal.h`** + **`src/portal.cpp`** (nomes em minúsculo, casando com
`matrices.h`, `textrendering.cpp`; `.h` entra no glob `include/*.h` do Makefile). Toda a lógica
de portais fica aqui; `main.cpp` apenas chama pontos de integração limpos. Reutiliza
`matrices.h` (`Matrix_*`, `crossproduct`, `dotproduct`, `norm`) e `glm::inverse`.

### `struct Portal` — um portal preso a uma superfície
Campos: `glm::vec3 center`, `normal` (unitária, apontando para **fora** da parede), `up`
(unitária, tangente à parede — define o "roll"), `float width, height`, `int colorId`
(`PORTAL_BLUE`/`PORTAL_ORANGE`), `Portal* link`.

Métodos:
- `glm::mat4 frame() const` — base ortonormal rígida `[right | up | normal | center]`
  (right = `crossproduct(up, normal)`). É o sistema de coordenadas local do portal **e a
  model matrix** para desenhar/estampar o quad.
- `glm::mat4 frameInverse() const` — inversa rígida (`[Rᵀ | −Rᵀ·center]`), via `glm::inverse`.

**Por que isto resolve "portais não paralelos às paredes":** o frame é construído puramente
a partir da `normal` dada; uma parede diagonal só fornece outra normal e tudo funciona sem
casos especiais.

### Factory de instanciação (a "developer experience" — requisito avaliado)
```cpp
// Cria UM portal sobre uma parede. upHint padrão = (0,1,0); se a normal for ~vertical
// (chão/teto), faz fallback de eixo. up = normaliza(upHint − proj na normal); right = up × normal.
static Portal Portal::onWall(glm::vec3 center, glm::vec3 normal,
                             glm::vec3 upHint = {0,1,0});

// Cria o PAR já vinculado (link cruzado) e atribui cores. UMA linha pro dev.

                                     float width = 1.2f, float height = 2.0f);
```
Uso no setup do `main()` (intuitivo, declarativo):
```cpp

    Portal::onWall({ x1,y1,z1 }, { nx1,ny1,nz1 }),   // azul
    Portal::onWall({ x2,y2,z2 }, { nx2,ny2,nz2 }));  // laranja
```

### `class PortalPair` — gestor do par (azul + laranja)
- **Núcleo matemático (público, reusado por render E física):**
  ```cpp
  // Matriz que leva o espaço do portal 'from' ao do 'to', com giro de 180° em torno
  // do 'up' de destino (sai-se de frente para fora do portal de destino):
  //   M = to.frame() * Matrix_Rotate(PI, up_to) * from.frameInverse()
  glm::mat4 transitionMatrix(const Portal& from, const Portal& to) const;
  glm::vec4 transformPoint (const Portal& from, glm::vec4 p) const; // w=1
  glm::vec4 transformVector(const Portal& from, glm::vec4 v) const; // w=0 (parte linear)
  ```
  Esta é a transformação linear pedida: serve para câmera virtual, velocidade, direção de
  olhar e (futuramente) o clone.
- **Travessia (SNAP):** mantém, por entidade, de que lado de cada plano ela estava no frame
  anterior. Se o centro cruzou do lado +normal para o −normal **dentro do retângulo** do
  portal neste frame, teleporta:
  ```cpp
  bool teleportIfCrossed(EntityXform& e); // pos, velocity(vec3), e (jogador) ângulos
  ```
- **Render (stencil, 1 nível):**
  ```cpp
  // Para cada portal: estampa o quad no stencil, calcula câmera virtual (transformPoint/Vector
  // sobre a câmera real), monta projeção e clip-plane do plano de destino, e chama drawScene.
  void renderViews(const glm::mat4& view, const glm::mat4& proj,
                   void (*drawScene)(), /* uniforms via PortalGL ctx */ int depth = 1);
  void renderSurfaces(); // desenha as molduras/anéis (emissivo azul/laranja)
  ```
  Para manter o módulo desacoplado de `main.cpp`, `renderViews` recebe os locations de uniform
  (`view`, `projection`, `clipPlane`) e o ponteiro de função `drawScene` num pequeno
  `struct PortalGL` configurado uma vez no início.

### Câmera virtual + clipping oblíquo
- **View virtual:** `view_virtual = view_real * transitionMatrix(dest→origem)` (renderiza-se a
  cena como vista a partir do portal de destino, equivalente a transformar a câmera real).
- **Clipping oblíquo:** para não desenhar geometria *entre* a câmera virtual e o portal de
  destino (tapando a vista), usar **`gl_ClipDistance[0]`** com um plano = plano do portal de
  destino em coordenadas de mundo. Escolhido em vez de mexer na matriz `-M*P` (Lengyel) por ser
  mais simples e seguro com a projeção custom do projeto. Default `clipPlane = (0,0,0,0)` →
  no-op; `glEnable(GL_CLIP_DISTANCE0)` só durante o render dos portais.
